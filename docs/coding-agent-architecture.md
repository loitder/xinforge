# Design: Xinforge Coding Agent Architecture

Generated by /office-hours on 2026-05-10
Branch: master
Repo: xinforge
Status: DRAFT
Mode: Builder

## Problem Statement

Xinforge should become a Codex / Claude Code style local coding agent. The goal is not just "LLM plus tools"; the product needs a durable agent runtime that can understand a repo, manage sessions, remember project facts, call models, execute tools, request approval for risky actions, edit files, run verification, and report diffs clearly.

The first serious version should use a complete internal architecture, even if only part of it is implemented initially. This keeps the prototype from growing into a tangled tool loop.

## What Makes This Cool

The coolest version is a local-first development partner:

- You open a repo and type `/implement add streaming output`.
- Xinforge reads the relevant code, builds a plan, tracks todos, uses memory, asks before risky actions, edits files, runs verification, and shows the final diff.
- You can quit and resume later without losing the task state.
- The product feels like a small agent operating system for software work, but the first build remains a CLI.

## Constraints

- Current codebase is a small Rust CLI with `agent.rs`, `llm.rs`, `tools.rs`, `skill.rs`, and `todo.rs`.
- Existing strengths: tool-calling loop, OpenAI API shape, file tools, bash, todo tracking, skill loading, subagent spawning, compaction.
- Missing foundations: persistent sessions, explicit permission gate, model abstraction, structured memory, diff/report layer, context builder, slash command router.
- First product target is self-use and learning, not enterprise multi-user deployment.
- The design should stay local-first. No daemon, hosted service, GUI, or marketplace in the first architecture milestone.

## Premises

1. A serious coding agent is built around state, not just prompts.
   Session, memory, todo state, tool results, approvals, and diffs must be first-class data.

2. Trust is a product feature.
   The agent must make file writes, shell commands, dependency changes, and git operations visible and confirmable.

3. Model providers should be replaceable.
   OpenAI is the default, but the rest of the agent should talk to a model runtime interface, not a hard-coded provider.

4. Context construction deserves its own module.
   The agent should decide what to send to the model each turn: system prompt, rules, project memory, session summary, recent messages, todos, files, git state, and loaded skills.

5. Subagents should begin as read-only workers.
   Parallel read-only exploration gives value without creating conflicting file edits.

6. The first durable product loop is `/implement`.
   It should turn a feature request into plan, exploration, approval, edits, verification, and diff summary.

## Approaches Considered

### Approach A: Minimal Usable Skeleton

Add only session persistence, slash commands, permissions, and diff reporting to the existing code.

Effort: M
Risk: Low

Pros:
- Fastest path to something useful.
- Reuses the current agent loop almost directly.
- Keeps implementation easy to understand.

Cons:
- Core responsibilities remain mixed inside `agent.rs`.
- Harder to add multi-model support, richer memory, or better tool policy later.
- Useful as a prototype, but likely needs a later refactor.

### Approach B: Layered Agent Runtime

Define a complete internal architecture: CLI, command router, session store, memory store, context builder, model runtime, tool registry, permission gate, workspace/git layer, planner, skill loader, subagent manager, compaction, and renderer.

Effort: L
Risk: Medium

Pros:
- Matches the shape of Codex / Claude Code style products.
- Gives every major concept a clear owner and data model.
- Lets Xinforge grow without turning the main loop into a pile of special cases.

Cons:
- More upfront design and refactoring.
- Requires disciplined interfaces before every feature feels visible to the user.
- Some modules will start thin, which can feel abstract if not tied to `/implement`.

### Approach C: Agent OS / Local Daemon

Build a long-running local service with CLI/TUI clients, persistent worker queues, shared memory, managed subagents, and background task execution.

Effort: XL
Risk: High

Pros:
- Highest long-term ceiling.
- Enables multiple frontends, background tasks, and richer orchestration.
- Could become a platform.

Cons:
- Too large for the current stage.
- Daemon lifecycle, concurrency, permissions, and crash recovery would dominate the build.
- Current code would mostly become a prototype rather than the foundation.

## Recommended Approach

Choose Approach B: Layered Agent Runtime.

Implementation should still proceed like Approach A: ship one vertical workflow first, then pull repeated responsibilities into the planned modules. The architecture should be complete; the first release should be narrow.

## Target Product Shape

Xinforge should support two entry styles:

```bash
xinforge
xinforge "explain this repository"
xinforge implement "add session persistence"
xinforge review
xinforge resume <session-id>
```

Inside the REPL, slash commands provide stable workflows:

```text
/implement <task>
/review
/diff
/status
/todo
/memory
/model
/skills
/compact
/resume <session-id>
/exit
```

Natural language remains supported, but core flows should use commands so they can be tested and improved.

## Core Components

### 1. CLI and Command Router

Owns user entrypoints, argument parsing, REPL input, slash command parsing, and dispatch.

Responsibilities:
- Start interactive mode.
- Run one-shot prompts.
- Dispatch slash commands.
- Attach commands to sessions.
- Route `/implement`, `/review`, `/resume`, `/diff`, `/memory`, and `/model`.

Suggested Rust module:

```text
src/cli.rs
src/commands.rs
```

Key types:

```rust
enum UserCommand {
    Chat(String),
    Implement(String),
    Review,
    Diff,
    Resume(String),
    Memory(MemoryCommand),
    Model(ModelCommand),
    Compact,
    Exit,
}
```

### 2. Session Store

Session is the durable record of a work episode. It should persist enough state that the agent can resume after exit or compaction.

Responsibilities:
- Create session ids.
- Append user, assistant, and tool messages.
- Store active todos.
- Store approvals.
- Store model config used for the session.
- Store summary after compaction.
- Store changed files and verification results.

Suggested paths:

```text
.xinforge/sessions/<session-id>/session.json
.xinforge/sessions/<session-id>/messages.jsonl
.xinforge/sessions/<session-id>/tools.jsonl
.xinforge/sessions/<session-id>/summary.md
.xinforge/sessions/<session-id>/diff.patch
```

Key data:

```rust
struct Session {
    id: String,
    created_at: DateTime,
    updated_at: DateTime,
    workdir: PathBuf,
    branch: Option<String>,
    model: ModelConfig,
    status: SessionStatus,
    summary: Option<String>,
    active_todos: Vec<TodoItem>,
}
```

### 3. Memory Store

Memory is cross-session knowledge. It should be explicit and inspectable, not hidden magic.

Three memory scopes:

- Project memory: repo-specific conventions, commands, architecture notes.
- User memory: user preferences across repos.
- Operational memory: discovered commands, environment quirks, repeated failure fixes.

Suggested paths:

```text
.xinforge/memory/project.md
.xinforge/memory/operations.jsonl
~/.xinforge/memory/user.md
```

Memory write policy:
- The agent can propose memory updates.
- The user confirms before durable user/project memory is written.
- Operational memory can be auto-suggested at task end.

Example project memory:

```markdown
# Project Memory

## Commands
- Test: `cargo test`
- Build: `cargo build`

## Style
- Prefer small Rust modules with explicit data structs.
- Use JSONL for append-only local state.
```

### 4. Model Runtime

The model runtime hides provider differences from the agent loop.

Responsibilities:
- Convert internal messages to provider format.
- Send tool schemas.
- Parse text, tool calls, finish reason.
- Handle provider-specific quirks.
- Track token usage when available.

Suggested modules:

```text
src/model/mod.rs
src/model/openai.rs
```

Core trait:

```rust
#[async_trait]
trait ModelProvider {
    async fn chat(&self, request: ChatRequest) -> anyhow::Result<ChatResponse>;
}
```

Important config:

```rust
struct ModelConfig {
    provider: String,
    model_id: String,
    base_url: Option<String>,
    api_key: String,
    max_tokens: u32,
    reasoning: ReasoningMode,
    temperature: Option<f32>,
}
```

### 5. Context Builder

Context builder decides what the model sees. This should be separate from model calling.

Inputs:
- System prompt.
- Developer rules.
- Project memory.
- User memory.
- Session summary.
- Recent messages.
- Active todos.
- Git status.
- Loaded skills.
- Relevant files.
- Tool definitions.

Responsibilities:
- Apply token budget.
- Prefer recent messages over stale detail.
- Include current todo state.
- Include only relevant memory.
- Preserve user decisions and approvals.

Suggested module:

```text
src/context.rs
```

Priority order:

1. System and safety rules.
2. Current user task.
3. Active todos and task state.
4. Recent conversation.
5. Session summary.
6. Project memory.
7. Relevant code snippets.
8. Tool result summaries.

### 6. Tool Registry

Tools need metadata, policy, schema, and executor in one place.

Responsibilities:
- Register available tools.
- Expose JSON schemas to the model.
- Execute approved tool calls.
- Normalize output.
- Truncate or summarize large outputs.
- Attach risk classification.

Suggested module:

```text
src/tools/mod.rs
src/tools/registry.rs
src/tools/fs.rs
src/tools/shell.rs
src/tools/git.rs
```

Tool metadata:

```rust
struct ToolSpec {
    name: String,
    description: String,
    schema: serde_json::Value,
    risk: ToolRisk,
    approval: ApprovalPolicy,
}
```

Risk levels:

```rust
enum ToolRisk {
    Read,
    Write,
    ShellSafe,
    ShellRisky,
    Network,
    Destructive,
}
```

Core tools:

- `read_file`
- `list_files`
- `search`
- `write_file`
- `edit_file`
- `apply_patch`
- `bash`
- `git_status`
- `git_diff`
- `todo`
- `load_skill`
- `spawn_subagent`
- `compact`

### 7. Permission Gate

The permission gate decides whether a tool call can run.

Responsibilities:
- Classify the action.
- Check user trust mode.
- Ask for confirmation when needed.
- Record approvals in the session.
- Block known dangerous actions.

Modes:

```text
read-only       Read tools only.
approval        Writes and risky commands require approval.
trusted         Writes and safe commands allowed; destructive still blocked.
dangerously-auto Explicit opt-in; still logs everything.
```

Policy examples:

- Read files: allow.
- Write project file: ask in approval mode.
- Run `cargo test`: allow or ask once per session.
- Run install command: ask.
- Git commit: ask.
- `rm -rf`, `git reset --hard`, force push: block or strong confirm.

### 8. Workspace and Git Layer

Coding agents need repo awareness.

Responsibilities:
- Detect repo root.
- Detect dirty worktree.
- Read branch and remotes.
- Produce status summaries.
- Capture before/after diff.
- Track files touched during a task.

Suggested module:

```text
src/workspace.rs
src/git.rs
```

Task final report should include:

- Files changed.
- Diff summary.
- Verification commands.
- Tests passed or failed.
- Remaining risks.

### 9. Planner and Todo

The planner owns task breakdown and progress tracking.

Responsibilities:
- Create todos for multi-step work.
- Mark one todo in progress.
- Update todo state after each action.
- Keep todos in session state.
- Render todos in the CLI.

Existing `todo.rs` is a good seed. It should become session-aware.

Todo item:

```rust
struct TodoItem {
    id: String,
    text: String,
    status: TodoStatus,
    source: TodoSource,
}
```

### 10. Skill Loader

Skills are specialized instructions loaded on demand.

Responsibilities:
- Load `skills/<name>/SKILL.md`.
- Show skill descriptions in context.
- Load full skill body only when needed.
- Track loaded skills in session.
- Support slash commands for manual skill loading.

Future frontmatter:

```yaml
---
name: rust-review
description: Review Rust code for correctness and idioms.
triggers:
  - rust
  - cargo
tools:
  - read_file
  - search
  - bash
---
```

### 11. Subagent Manager

Subagents should begin as isolated read-only workers.

Responsibilities:
- Spawn a subagent with a scoped prompt.
- Restrict tool access.
- Capture logs and final summary.
- Attach subagent output to the parent session.
- Support list/status/kill later.

First use cases:

- Find relevant files for a feature.
- Review a proposed implementation plan.
- Inspect test commands.
- Summarize a module.

Initial policy:

- Subagents get read/search/git status tools.
- Subagents do not write files.
- Main agent owns edits.

### 12. Diff and Patch Layer

Diff is the user trust surface.

Responsibilities:
- Capture pre-task git status.
- Capture post-task git diff.
- Summarize changed files.
- Show patch snippets.
- Save patch to session.

Suggested commands:

```text
/diff
/diff --stat
/diff --file src/agent.rs
```

Final report format:

```text
Done: added session persistence.

Changed:
- src/session.rs: new session store
- src/agent.rs: loads and appends messages
- README.md: documents /resume

Verified:
- cargo test: passed

Risk:
- session migration not needed yet because this is first schema
```

### 13. Compaction

Current compaction should become a session feature.

Three levels:

- Micro compaction: summarize old tool outputs.
- Session summary: summarize old messages into `summary.md`.
- Transcript archive: preserve raw JSONL on disk.

Summary must preserve:

- User goal.
- Decisions made.
- Active todos.
- Files changed.
- Commands run.
- Tests and failures.
- Open risks.

### 14. Renderer

Renderer controls terminal output. It should keep the product readable.

Output types:

- Assistant text.
- Tool call start.
- Tool result summary.
- Approval prompt.
- Todo list.
- Error block.
- Diff summary.
- Final report.

First version can use plain ANSI output. Keep formatting centralized so a TUI can replace it later.

## Golden Workflow: `/implement`

`/implement <task>` is the first full product loop.

Flow:

1. Create or resume a session.
2. Record user task.
3. Build initial context with project memory and git state.
4. Ask model to produce a plan.
5. Store plan as todos.
6. Run read/search tools to inspect code.
7. Optionally spawn read-only subagents for parallel exploration.
8. Present implementation plan if edits are non-trivial.
9. Ask for approval before writes.
10. Apply edits through patch/file tools.
11. Run verification commands, with approval if needed.
12. Capture git diff.
13. Produce final report.
14. Offer memory updates if useful.

## Milestones

### M1: Session and Commands

- Add `.xinforge/sessions`.
- Persist messages and todos.
- Add command router.
- Implement `/implement`, `/resume`, `/status`, `/todo`.

### M2: Permission and Diff

- Add tool risk metadata.
- Add approval mode.
- Gate file writes and risky shell commands.
- Add `git_status` and `git_diff` tools.
- Add `/diff`.

### M3: Context Builder and Memory

- Move prompt assembly out of `agent.rs`.
- Add project memory and user memory.
- Include memory and session summary in context.
- Add `/memory`.

### M4: Model Runtime

- Extract provider trait.
- Move current OpenAI call into provider adapter.
- Support model profiles in config.
- Add `/model`.

### M5: Subagent Manager

- Make subagent runs stateful.
- Restrict subagents to read-only tools first.
- Store subagent logs and summaries in session.
- Add `/agents` later if needed.

## Success Criteria

The architecture is successful when this works end to end:

```text
/implement add a /diff command
```

Expected behavior:

- Xinforge creates a session.
- It inspects the repo.
- It creates todos.
- It asks before writing files.
- It edits the code.
- It runs `cargo test` or asks before running it.
- It shows changed files and diff summary.
- It can resume the same session later.
- It records useful project memory only with approval.

## Open Questions

- Should the first persistent format be JSONL files only, or SQLite later?
- Should user memory live only in `~/.xinforge`, or should project memory be committed?
- Should `/implement` always ask before the first write, or support trusted project mode?
- Should model config live in env vars only, or a `.xinforge/config.toml` file?
