# Design: Xinforge Model Provider Runtime

Generated by /office-hours on 2026-05-10
Branch: master
Repo: xinforge
Status: DRAFT
Mode: Builder

## Problem Statement

Xinforge needs a model layer that gives the agent one stable OpenAI provider
boundary. Today the agent calls `src/llm.rs` directly, using OpenAI chat JSON.
That is fine for the prototype, but it means the agent loop owns provider
details, message details, tool-call parsing, and response shape.

The next design should extract this into an independent crate, tentatively named
`xinforge-model`. The crate should handle model providers, streaming responses,
multi-turn message structures, tool schemas, tool-call deltas, provider quirks,
usage accounting, and fallback policy. The agent should submit one internal
request format and consume one internal event stream.

## What Makes This Cool

The cool version is that Xinforge stops coupling itself to provider APIs and
starts working in agent events:

```text
User message in
  -> model stream starts
  -> text delta
  -> tool call delta
  -> tool call complete
  -> usage reported
  -> response complete
```

That lets the CLI render live output, execute tools as soon as calls are complete,
switch providers without touching the agent loop, and later use different models
for planning, editing, summarizing, and subagents.

## Constraints

- Current provider code lives in `src/llm.rs`.
- Current request shape is OpenAI chat completions.
- Current agent loop expects a blocking `chat(...) -> (text, tool_calls, reasoning, finish_reason)`.
- Current tools are described as OpenAI function tools in `src/tools.rs`.
- Current conversation messages are raw `serde_json::Value`.
- The first implementation should not rewrite the whole agent.
- The crate should be usable outside Xinforge, so it should avoid depending on
  agent, tool execution, session store, workspace, or CLI rendering code.

## Premises

1. Provider APIs are not the agent contract.
   The agent should speak to `xinforge-model` through provider-neutral types,
   even while the first implementation only ships the OpenAI provider.

2. Streaming is the primary API, not an optional feature.
   Non-streaming responses can be implemented by collecting the stream. The
   agent runtime and CLI should be designed around events.

3. Tool calls need a provider-neutral intermediate representation.
   Each provider has different tool-call chunks, argument deltas, ids, and
   finish markers. The runtime owns the state machine that turns those chunks
   into complete `ToolCall` values.

4. Conversation management belongs partly in the model crate, but session
   persistence does not.
   `xinforge-model` owns message types, response assembly, and provider
   transcript conversion. `xinforge-agent` owns saved sessions, compaction,
   project memory, and context selection.

5. Model selection should be explicit before it becomes automatic.
   First ship named model profiles and a deterministic fallback policy. Later
   add task-based automatic routing once real usage data exists.

## Approaches Considered

### Approach A: Provider Adapter Crate

Summary: Extract `src/llm.rs` into a small crate with a `ModelProvider` trait and
one OpenAI adapter.

Effort: M
Risk: Low

Pros:
- Fastest path away from hard-coded provider calls.
- Reuses nearly all existing request and response parsing logic.
- Easy to integrate without changing the agent loop much.

Cons:
- Does not solve model profiles, fallback, capability metadata, or usage policy.
- Leaves tool-call streaming and event assembly under-designed.
- Likely needs another refactor once `/model` and subagent model selection arrive.

Reuses:
- `src/llm.rs`
- `src/tools.rs` tool schema format
- `Agent::agent_loop` request call site

### Approach B: Model Runtime Crate

Summary: Create `xinforge-model` as a real runtime crate with provider registry,
model profiles, provider-neutral messages, streaming events, tool-call assembly,
capabilities, usage reporting, and fallback policy.

Effort: L
Risk: Medium

Pros:
- Gives the agent one stable interface for every provider.
- Makes streaming and tool calls first-class instead of bolt-ons.
- Creates the foundation for `/model`, subagent model assignment, budgets, and
  future automatic model routing.

Cons:
- Requires careful type design before the implementation pays off.
- More initial code than a direct `llm.rs` extraction.
- Provider adapters must resist leaking provider-specific fields into core types.

Reuses:
- Existing OpenAI request body construction.
- Existing message and tool-call tests as adapter regression tests.
- Existing architecture doc milestone M4.

### Approach C: Full Model Orchestrator

Summary: Build an automatic model router that chooses models per task, monitors
cost, tracks failure rates, and switches between fast, cheap, and strong models.

Effort: XL
Risk: High

Pros:
- Highest long-term ceiling.
- Enables patterns like cheap model for search, strong model for design, fast
  model for summaries.
- Makes Xinforge feel more autonomous.

Cons:
- Premature without usage data and evals.
- Adds policy complexity before the provider abstraction is stable.
- Hard to debug when the wrong model gets selected.

Reuses:
- Future capability metadata from Approach B.

## Recommended Approach

Choose Approach B: Model Runtime Crate.

Build the architecture as a runtime crate, but implement the first slice narrowly:
OpenAI streaming, OpenAI profile, blocking compatibility helper,
tool-call assembly, usage fields, and a simple named-profile selector. That gives
Xinforge a clean boundary now without making automatic routing a dependency.

## Crate Shape

Use a workspace layout:

```text
crates/
  xinforge-model/
    Cargo.toml
    src/
      lib.rs
      error.rs
      message.rs
      request.rs
      response.rs
      stream.rs
      tool.rs
      usage.rs
      profile.rs
      provider/
        mod.rs
        openai.rs
      runtime.rs
```

Keep the main binary in the existing root crate at first. Later, the root can
also become a workspace package if the project wants more crates.

## Core API

The agent should depend on these types:

```rust
pub struct ModelRuntime {
    registry: ProviderRegistry,
    profiles: ModelProfileStore,
}

impl ModelRuntime {
    pub async fn stream_chat(
        &self,
        profile: &ProfileRef,
        request: ChatRequest,
    ) -> Result<ChatEventStream, ModelError>;

    pub async fn chat(
        &self,
        profile: &ProfileRef,
        request: ChatRequest,
    ) -> Result<ChatResponse, ModelError>;
}
```

`chat` is a collector over `stream_chat`, used to migrate the current agent loop.
New agent code should prefer `stream_chat`.

## Provider Trait

Provider adapters implement the low-level protocol boundary:

```rust
#[async_trait::async_trait]
pub trait ModelProvider: Send + Sync {
    fn id(&self) -> ProviderId;

    fn capabilities(&self, model_id: &str) -> ModelCapabilities;

    async fn stream_chat(
        &self,
        profile: &ModelProfile,
        request: ChatRequest,
    ) -> Result<ProviderEventStream, ModelError>;
}
```

Provider adapters may emit provider events, but `ModelRuntime` normalizes them
into public `ChatStreamEvent` values before the agent sees them.

## Messages

Replace raw JSON messages with typed messages:

```rust
pub enum Message {
    System(SystemMessage),
    Developer(DeveloperMessage),
    User(UserMessage),
    Assistant(AssistantMessage),
    Tool(ToolMessage),
}

pub struct AssistantMessage {
    pub content: Vec<ContentBlock>,
    pub tool_calls: Vec<ToolCall>,
    pub reasoning: Option<ReasoningTrace>,
    pub provider_meta: ProviderMeta,
}

pub enum ContentBlock {
    Text(String),
    Image(ImageRef),
}
```

First version can support text-only user messages. The enum keeps the API ready
for screenshots and multimodal models without redesigning the transcript.

## Requests

```rust
pub struct ChatRequest {
    pub messages: Vec<Message>,
    pub tools: Vec<ToolSpec>,
    pub tool_choice: ToolChoice,
    pub max_output_tokens: Option<u32>,
    pub temperature: Option<f32>,
    pub reasoning: ReasoningConfig,
    pub metadata: RequestMetadata,
}
```

Rules:

- `ChatRequest` is provider-neutral.
- Provider adapters decide which fields are supported.
- Unsupported fields become explicit warnings or errors based on policy.
- Agent context construction happens before this request is created.

## Stream Events

Streaming should be the canonical output:

```rust
pub enum ChatStreamEvent {
    Started {
        request_id: RequestId,
        provider: ProviderId,
        model: String,
    },
    ReasoningDelta {
        text: String,
    },
    TextDelta {
        text: String,
    },
    ToolCallDelta {
        index: usize,
        id: Option<String>,
        name: Option<String>,
        arguments_delta: String,
    },
    ToolCallReady {
        call: ToolCall,
    },
    Usage {
        usage: TokenUsage,
    },
    Finished {
        reason: FinishReason,
        response: ChatResponse,
    },
    Warning {
        warning: ModelWarning,
    },
}
```

The important event is `ToolCallReady`. The agent should not parse partial JSON
arguments. The model runtime owns partial argument buffering and emits a ready
tool call only when arguments parse successfully.

## Tool Representation

```rust
pub struct ToolSpec {
    pub name: String,
    pub description: String,
    pub input_schema: serde_json::Value,
}

pub struct ToolCall {
    pub id: ToolCallId,
    pub name: String,
    pub arguments: serde_json::Value,
}

pub struct ToolResult {
    pub tool_call_id: ToolCallId,
    pub content: Vec<ContentBlock>,
    pub is_error: bool,
}
```

Provider adapters convert `ToolSpec` into their own schema format. The agent
continues to own tool execution. The model crate only describes requested calls
and formats returned tool results as messages.

## Tool-Call Assembly State Machine

The runtime needs a small state machine per streamed response:

```text
empty
  -> seen tool call index
  -> name known
  -> id known
  -> arguments buffer appended
  -> finish marker or parse success
  -> ToolCallReady
```

Rules:

- Buffer arguments by provider call index, not only provider id. Some providers
  do not send an id until late.
- Emit `ToolCallReady` exactly once per call.
- If argument JSON is invalid at finish, emit `ModelError::InvalidToolArguments`
  with the raw argument buffer.
- Preserve provider raw chunks in debug mode only, not in the public happy path.

## Responses

```rust
pub struct ChatResponse {
    pub id: Option<String>,
    pub model: String,
    pub provider: ProviderId,
    pub message: AssistantMessage,
    pub finish_reason: FinishReason,
    pub usage: Option<TokenUsage>,
    pub warnings: Vec<ModelWarning>,
}

pub enum FinishReason {
    Stop,
    Length,
    ToolCalls,
    ContentFilter,
    Error,
    Unknown(String),
}
```

The blocking `chat` helper collects text, tool calls, finish reason,
usage, and warnings from the stream.

## Profiles

Model profiles are user-facing model configurations:

```rust
pub struct ModelProfile {
    pub name: String,
    pub provider: ProviderId,
    pub model_id: String,
    pub base_url: Option<String>,
    pub api_key: String,
    pub defaults: GenerationDefaults,
    pub capabilities: ModelCapabilities,
    pub fallback: Vec<String>,
}
```

Example config:

```toml
[profiles.openai-default]
provider = "openai"
model_id = "gpt-5.5"
base_url = "https://api.openai.com/v1"
api_key = "<resolved by caller>"
max_output_tokens = 8000
reasoning = "auto"
fallback = ["openai-gpt-5-mini"]

[profiles.openai-gpt-5-mini]
provider = "openai"
model_id = "gpt-5-mini"
base_url = "https://api.openai.com/v1"
api_key = "<resolved by caller>"
max_output_tokens = 8000

```

The current environment variables remain supported as an implicit default profile:

```text
OPENAI_BASE_URL
OPENAI_API_KEY
MODEL_ID
```

That keeps quick start working.

## Capabilities

```rust
pub struct ModelCapabilities {
    pub streaming: bool,
    pub tools: bool,
    pub parallel_tool_calls: bool,
    pub reasoning: ReasoningSupport,
    pub vision: bool,
    pub max_context_tokens: Option<u32>,
    pub max_output_tokens: Option<u32>,
}
```

Capabilities should inform validation. If a request includes tools for a model
without tool support, the runtime should fail before making the provider request.

## Fallback Policy

First version fallback should be simple and deterministic:

```rust
pub enum FallbackMode {
    Disabled,
    OnProviderError,
    OnRateLimit,
    OnTimeout,
}
```

Do not fallback on semantic failures, weak answers, invalid edits, or bad plans.
Those are agent-level quality problems. Fallback should handle infrastructure and
provider availability problems only.

When fallback happens, emit a `Warning` event and preserve the final provider and
model in `ChatResponse`.

## Provider Adapters

### OpenAI Adapter

Responsibilities:

- Convert `Message` into OpenAI chat messages.
- Convert `ToolSpec` into `tools`.
- Send `/chat/completions` requests.
- Support `stream: true` SSE parsing.
- Parse text deltas, tool call deltas, finish reason, and usage.
- Support non-streaming through the same response assembly path.
- Handle OpenAI provider finish reasons or response quirks.

## Conversation Management

`xinforge-model` should provide helpers for transcript construction:

```rust
pub struct Conversation {
    messages: Vec<Message>,
}

impl Conversation {
    pub fn push_user_text(&mut self, text: impl Into<String>);
    pub fn push_assistant(&mut self, response: ChatResponse);
    pub fn push_tool_result(&mut self, result: ToolResult);
    pub fn messages(&self) -> &[Message];
}
```

This is not session storage. It is an in-memory typed transcript. The agent can
serialize its own session format or use serde on these types.

## Agent Integration

Current call site:

```rust
let (text, tool_calls_content, finish_reason) = llm::chat(...).await?;
```

Migration call site:

```rust
let response = model_runtime
    .chat(&profile_ref, chat_request)
    .await?;
```

Streaming call site:

```rust
let mut stream = model_runtime
    .stream_chat(&profile_ref, chat_request)
    .await?;

while let Some(event) = stream.next().await {
    match event? {
        ChatStreamEvent::TextDelta { text } => renderer.write_delta(&text),
        ChatStreamEvent::ReasoningDelta { text } => renderer.write_reasoning(&text),
        ChatStreamEvent::ToolCallReady { call } => agent.enqueue_tool_call(call),
        ChatStreamEvent::Finished { response, .. } => agent.record_response(response),
        _ => {}
    }
}
```

First integration should use blocking `chat` to reduce risk. Then update the CLI
renderer and agent loop to consume streaming events.


Do not add automatic routing to `/model` yet. Keep the command focused on
visibility, selection, validation, and testing.

## Error Model

```rust
pub enum ModelError {
    MissingApiKey,
    UnknownProvider { provider: String },
    UnknownProfile { profile: String },
    UnsupportedCapability { capability: String, model: String },
    ProviderHttp { status: u16, body: String },
    RateLimited { retry_after_ms: Option<u64> },
    Timeout,
    StreamParse { detail: String },
    InvalidToolArguments { tool_name: String, raw: String },
    Serialization { detail: String },
}
```

Errors should be specific enough for the agent to decide whether fallback is
allowed and for the CLI to show an actionable message.

## Testing Strategy

Unit tests:

- Message conversion into OpenAI request JSON.
- Tool schema conversion.
- Streaming text delta assembly.
- Streaming tool argument delta assembly.
- Invalid tool argument failure.
- Finish reason mapping.
- Usage parsing.
- Missing API key validation.

Fixture tests:

- Save provider SSE chunks as fixture files.
- Feed fixtures into adapter parser.
- Assert final `ChatResponse` and intermediate events.

Integration tests:

- Mock HTTP server returns OpenAI streaming response.
- Mock HTTP server returns tool-call streaming response.
- Runtime fallback from one failing mock provider to one healthy provider.

No live provider tests should run by default.

## Milestones

### M1: Core crate and blocking compatibility

- Create `crates/xinforge-model`.
- Add typed `Message`, `ToolSpec`, `ToolCall`, `ChatRequest`, `ChatResponse`.
- Implement `ModelProvider` trait.
- Move current OpenAI blocking call into adapter.
- Keep current agent behavior working through `ModelRuntime::chat`.

### M2: Streaming and tool-call assembly

- Add stream event types.
- Implement OpenAI SSE parser.
- Implement tool-call assembly state machine.
- Add fixture tests for text, and tool calls.

### M3: Profiles and `/model`

- Add profile loading from env and config file.
- Add provider registry.
- Add `/model list`, `/model use`, `/model show`, `/model test`.
- Store selected profile in session once session storage exists.

### M4: Fallback

- Add deterministic fallback for provider errors, rate limits, and timeouts.
- Emit warnings when fallback happens.

### M5: Agent streaming integration

- Update agent loop to consume `ChatStreamEvent`.
- Render text deltas live.
- Execute tool calls after `ToolCallReady`.
- Record final `ChatResponse` into session.

## Success Criteria

This design is successful when these work:

```text
xinforge
>> /model list
>> /model use openai-default
>> explain src/agent.rs
```

Expected behavior:

- Xinforge shows current profile and capabilities.
- The agent sends typed messages through `xinforge-model`.
- The provider streams text deltas.
- Tool calls arrive as completed `ToolCall` values.
- The old blocking agent loop can still run during migration.

## Open Questions

- Should `xinforge-model` expose serde types as stable public API from day one,
  or keep serialization behind feature flags until the transcript format settles?
- Should profile config live in `.xinforge/config.toml`, `~/.xinforge/config.toml`,
  or both with project overriding user defaults?
- Should tool-call streaming allow early execution before the assistant finish
  reason arrives, or wait for the whole model turn to finish in v1?
- Should provider adapters preserve raw provider payloads in debug logs, and if
  so, how do we avoid leaking secrets into local transcripts?

## Next Steps

1. Create the `xinforge-model` crate with core types and a blocking adapter.
2. Replace `src/llm.rs` call sites through a compatibility shim.
3. Add streaming fixture tests before changing the agent loop.
4. Add `/model` only after profiles exist.
5. Defer automatic model routing until the runtime has real usage data.

## What I Noticed

- You did not ask for "support more models"; you asked for a provider management
  layer with streaming, multi-turn conversation, and tools. That is the right
  boundary for an agent runtime.
- The existing architecture doc already pointed at M4, but this design makes M4
  independently shippable as a crate instead of a local refactor.
- The key taste call is making streaming the primary API. That keeps the runtime
  honest and avoids building a blocking abstraction that has to be broken later.
